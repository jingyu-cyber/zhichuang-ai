# 智创Agent 开发 SOP

适用范围：智创Agent Web 首版、后端服务、RAG 知识库、LangGraph 多 Agent 工作流、项目代码分析、教师学情诊断看板，以及后续鸿蒙端客户端。

## 1. 开发原则

1. 先保证学校真实使用场景，再包装比赛展示。
2. P0 主线固定为：学生端成长闭环、项目代码分析、教师学情诊断看板、RAG 可追溯知识库。
3. 教师端不是审核流。教师直接查看系统生成的班级项目情况、项目报告、能力分布和共性问题。
4. 模型能力通过 Provider Adapter 接入，业务代码不绑定某一个模型厂商。
5. 长任务和多 Agent 编排放进 LangGraph，确定性业务规则放进普通 service。
6. 评分是基于证据的相对画像，不表达对学生绝对能力的最终判断。

## 2. 角色与权限

| 角色 | 核心权限 |
| --- | --- |
| 学生 | 查看个人画像、学习计划、项目分析报告、推荐资源、竞赛建议 |
| 教师 | 查看授权课程和班级的项目分析看板、学生项目报告、班级能力分布 |
| 管理员 | 管理课程、班级、知识库资料、Rubric、演示数据和系统配置 |

权限规则以课程、班级、角色为边界。公网 Demo 使用演示账号和示例课程数据。

## 3. 分支与任务流

1. `main`：稳定主线，要求可运行、可演示。
2. `feature/<scope>-<name>`：功能开发，例如 `feature/assignment-report`。
3. `fix/<scope>-<name>`：缺陷修复，例如 `fix/rag-citation`。
4. `docs/<name>`：文档、PPT、方案调整。

每个任务进入开发前，先写清楚：

- 用户场景：学生、教师或管理员要完成什么事。
- 输入输出：用户提交什么，系统返回什么。
- 验收标准：接口、页面、报告或评测记录如何判断完成。
- 影响范围：前端、后端、RAG、数据库、LangGraph、部署。

## 4. 本地开发流程

首次准备：

```bash
make init
```

后端：

```bash
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

涉及 LangGraph 或 RAG 开发时安装额外依赖：

```bash
pip install -e ".[ai,rag,dev]"
```

前端：

```bash
cd frontend
npm install
npm run dev
```

Docker 联调：

```bash
docker compose up --build
```

常用检查：

```bash
make lint
make test
make check
```

## 5. 需求到代码的交付流程

1. 写需求卡片：说明场景、角色、输入输出和优先级。
2. 定接口：先在 `docs/API设计.md` 中补接口草案。
3. 定数据：涉及持久化时同步更新 `docs/数据模型.md`。
4. 后端实现：先 schema，再 service，再 route；长任务再接 graph。
5. 前端实现：先页面状态和接口封装，再补可视化和交互细节。
6. 测试验证：至少覆盖 service 或 API 层，复杂 Agent 流程要留评测样例。
7. 更新文档：README、SOP、架构约定或 Roadmap 有变化时同步改。

## 6. 后端开发规范

后端分层：

| 层 | 目录 | 约定 |
| --- | --- | --- |
| API | `backend/app/api` | 只处理 HTTP、参数校验、响应模型 |
| Schema | `backend/app/schemas` | Pydantic 请求和响应结构 |
| Service | `backend/app/services` | 确定性业务逻辑、权限检查、报表聚合 |
| Graph | `backend/app/ai/graphs` | 多 Agent、长任务、中断恢复、状态流转 |
| Provider | `backend/app/ai/providers` | 大模型、Embedding、Rerank 接入适配 |
| RAG | `backend/app/rag` | 解析、切分、入库、检索、引用校验 |
| Model | `backend/app/models` | SQLAlchemy ORM 模型 |

接口返回需要稳定、可解释、便于前端渲染。不要把 Prompt 输出原文直接当业务结构，必须经过 schema 校验。

## 7. 前端开发规范

首版 Web 使用 React + TypeScript + Vite + Ant Design + ECharts。

目录建议：

| 目录 | 职责 |
| --- | --- |
| `src/app` | 应用入口、路由、全局 Provider |
| `src/pages` | 页面级组件 |
| `src/features` | 按业务模块拆分的功能组件 |
| `src/shared/api` | API client 和请求封装 |
| `src/shared/components` | 可复用 UI 组件 |
| `src/shared/types` | 前端共享类型 |

页面优先服务高频工作流：

1. 学生上传项目并查看报告。
2. 教师查看课程项目班级看板。
3. 管理员维护知识库资料和 Rubric。
4. 学生查看学习路径、竞赛建议、组队推荐。

## 8. LangGraph 工作流 SOP

适合进入 LangGraph 的任务：

- 项目代码分析：解析代码、检查结构、评估质量、抽取能力证据、生成报告。
- 学生计划生成：目标澄清、画像读取、资源检索、阶段计划、用户确认后修改计划。
- 竞赛推荐：画像匹配、资料检索、可行性评估、风险提示、推荐解释。
- 组队推荐：能力互补、方向匹配、候选解释、用户确认后保存。
- 定期复盘：读取任务进展、总结偏差、更新路径、生成下一阶段建议。

不适合进入 LangGraph 的任务：

- 登录、权限判断、数据库 CRUD。
- 简单统计查询。
- 确定性评分汇总。
- 前端页面状态切换。

Graph 输出必须满足：

1. 有结构化状态。
2. 有可恢复的任务 ID。
3. 有引用来源或证据说明。
4. 有失败状态和重试策略。
5. 最终输出能落库，供学生端和教师端复用。

## 9. RAG 知识库建设 SOP

首批知识库资料作为需求列入项目范围，重点围绕三条路径：

1. 算法竞赛：竞赛介绍、报名规则、训练路线、经典题单、常见算法专题。
2. AI 应用开发：大模型应用、RAG、Agent、Prompt、评测、部署案例。
3. 软件项目实践：需求分析、架构设计、接口设计、数据库设计、测试、项目文档。

首版资料范围：

- 至少 5 门课程资料。
- 至少 10 条竞赛资料。
- 至少 10 个项目案例。
- 配套 Rubric、评分口径、报告模板和引用来源。

资料入库流程：

1. 登记资料来源、标题、类型、适用路径、课程或竞赛标签。
2. 解析文档，按章节、知识点、任务步骤切分。
3. 生成 embedding，写入向量库。
4. 建立元数据：课程、章节、难度、更新时间、来源类型。
5. 抽样检索验证，记录命中片段和引用质量。
6. 进入知识库管理页面供管理员维护。

## 10. 项目代码分析 SOP

输入：

- 项目标题、课程、班级、提交学生。
- 项目要求、评分 Rubric、代码压缩包或仓库链接。
- 可选：README、运行说明、测试用例、报告文档。

处理流程：

1. 文件安全检查和项目结构解析。
2. 识别语言、框架、依赖、入口文件和测试文件。
3. 结合 Rubric 分析功能完成度。
4. 分析代码结构、可读性、异常处理、模块边界、测试覆盖。
5. 提取能力证据：算法、工程实践、问题拆解、调试、文档表达。
6. 生成学生个人项目报告。
7. 聚合班级分布、共性问题和讲评建议，供教师看板展示。

输出：

- 项目报告 ID。
- 多维度评分和证据。
- 主要问题和改进建议。
- 教师端班级统计和共性问题摘要。

## 11. Prompt 与评测 SOP

所有核心 Prompt 需要沉淀为版本化资产：

- 任务名称。
- 输入 schema。
- 输出 schema。
- Prompt 模板。
- 引用规则。
- 失败样例。
- 评测样例。

每次调整 Prompt 或 Agent 流程后，至少验证：

1. 是否能按 schema 输出。
2. 是否能引用知识库来源。
3. 是否出现编造链接、编造竞赛时间、编造课程事实。
4. 是否能区分“证据不足”和“能力不足”。
5. 是否能给出学生可执行建议和教师可读统计说明。

## 12. 测试与验收

后端：

- API 健康检查。
- schema 校验。
- service 单元测试。
- RAG 检索命中测试。
- Agent graph 流程测试。

前端：

- 页面可访问。
- API 错误态和加载态。
- 表格、图表、报告详情页基本渲染。
- 移动端关键页面布局。

交付验收：

- 公网 Demo 可访问。
- 演示账号可登录。
- 示例项目可生成报告。
- 教师看板可查看班级统计。
- 知识库答案带引用来源。

## 13. 提交与合并规范

提交信息建议：

```text
feat(assignment): add report summary api
fix(rag): handle empty citation list
docs(sop): add knowledge ingestion workflow
chore(repo): add editorconfig
```

PR 描述至少包含：

- 改了什么。
- 如何验证。
- 是否影响接口、数据模型或部署。
- 截图或接口示例，如果涉及前端。

合并前必须确认：

1. 不提交 `.env`、本地数据库、上传文件、依赖目录和构建产物。
2. `make check` 或等价检查已执行。
3. 文档和代码保持一致。

## 14. 部署 SOP

开发期：

- SQLite + ChromaDB 本地开发。
- FastAPI + Vite 本机联调。

公网 Demo：

- 后端容器化部署。
- 前端静态构建后部署。
- PostgreSQL + pgvector。
- Redis 用于任务队列、缓存和长任务状态。
- 使用演示账号和示例课程数据。

学校部署：

- 统一身份认证可作为后续集成项。
- 课程、班级、角色权限进入正式配置。
- 日志、任务状态、报告记录需要持久化。

## 15. 每日开发节奏

1. 开始前：确认今天任务对应哪个 P0/P1 场景。
2. 开发中：接口、schema、页面状态同步推进。
3. 收尾前：跑检查，补文档，记录未完成事项。
4. 演示前：准备固定样例、固定账号、固定讲解路径。
