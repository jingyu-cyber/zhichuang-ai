背景

Why Now

计算机人才培养正在从“学过什么”转向“做成什么”

课程项目能反映工程实践，但提交后往往只留下分数，难形成长期能力证据。
竞赛双创需要方向判断、项目积累和组队协作，学生常常不知道从哪里开始。
教师指导教师需要快速看到班级项目共性问题，而不是逐份手动阅读材料。
机会点

用项目分析把学生提交物转化为结构化报告，再把报告反哺画像、路径、竞赛和组队。

项目资产 -> 能力证据 -> 个性化成长 -> 教学诊断
项目材料是最可靠的能力入口
这里要把学生和教师的共同痛点合在一起：学生需要成长证据，教师需要学情洞察，而项目材料正好是二者之间的连接点。

项目分析

Analysis Pipeline

项目分析不是一次回复，而是多阶段工作流

Parse文件解析

限制文件类型与大小，识别可分析文本、目录、入口文件、依赖。

Structure结构摘要

识别框架、模块边界、配置文件、测试文件、文档完整度。

Review质量评估

分析可读性、复杂度、异常处理、重复代码、工程规范。

Evidence能力证据

提取能支撑算法、工程实践、AI 应用、表达协作等维度的证据。

Report报告生成

生成多维评分、问题、亮点、改进任务和教师看板聚合字段。

多 Agent 拆分，让分析过程可追踪
这页回应用户之前提到的关键问题：上传代码一次对话不够，内部需要拆成多个分析节点，每个节点产生结构化状态和证据。

RAG

Trust Mechanism

RAG 不只是检索，而是可信回答机制

Ingest资料入库

登记标题、类型、适用路径、标签、来源链接和更新记录。

Chunk切片建索引

按章节、知识点、任务步骤切分，写入向量库与元数据。

Retrieve语义检索

根据用户问题、角色和场景检索课程、竞赛或项目案例片段。

Answer带引用回答

生成回答时展示来源标题、路径、更新时间和片段。

Guard边界说明

资料不足时明确说明无法确认，不编造竞赛时间、课程事实或链接。

可信回答的关键是引用和不确定性表达
RAG 这一页不要只讲技术组件，要讲用户为什么信它：能看到来源，能知道更新时间，资料不足时不会硬编。

预期成效

Expected Outcome

让学生项目成为教学与成长的共同证据

对学生

把课程项目、竞赛作品、个人作品转化为能力证据，获得更清晰的学习路径、竞赛准备和组队建议。

对教师

从批量学生项目中快速看到共性问题、单个学生报告、班级能力分布和讲评建议。

对学院

沉淀计算机学科知识库、项目案例库和教学数据资产，支撑人才培养质量提升。

对作品

形成可运行、可验证、可扩展的学科垂类智能体应用，而不是一次性展示页面。

从“学生提交一个项目”开始，让教学过程产生可解释、可追踪、可复用的能力证据。
智创Agent·计算机学科垂类大模型与双创能力赋能平台
收束时回到一句话价值：学生项目不只是交付物，它是学生成长、教师诊断和学院数据资产的共同入口。
